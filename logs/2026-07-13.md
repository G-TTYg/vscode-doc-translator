### 目标

在开始实现前，把用户的初步想法整理成可维护、可继续迭代的项目本地设计文档。

### 上下文 / 发现

- 仓库只有 `.git`，没有 source、README、AGENTS、docs、tests 或 package 配置。
- 用户希望做一个 VS Code 文档翻译插件，触发方式包括右键菜单和 CLI。
- 用户希望支持 Google Translation API、DeepL API、Microsoft Translator 和 OpenAI-compatible LLM API。
- 用户希望译文保存在源文件同目录，并能作为普通文件被 VS Code 打开。
- 用户希望同目录存在默认隐藏的 cache/metadata，命名清晰，包含源文档 hash、译文 hash、译文文件名和新鲜度判断信息。
- 用户补充确认：传统机器翻译按分段 units 处理；AI/LLM 翻译要获得按原文顺序排列、带 id 的全文 JSON 上下文，并输出对应 id 的扁平 JSON 译文结果。
- 用户补充确认：译文默认可见地放在源文件旁，方便直接打开。
- 用户补充确认：Markdown 自动翻译注释头可以做成设置项，但 metadata JSON 内必须保留完整权威信息；隐藏目录名要更专用，避免和项目目录冲突。

### 决策

- VS Code 插件和 CLI 共用 TypeScript core。
- Google/DeepL/Microsoft/OpenAI-compatible 通过 provider adapter 接入。
- 文件类型通过 format adapter 处理解析和重建。
- 传统机器翻译使用分段 units；AI/LLM 使用 ordered JSON context，并返回扁平 JSON 译文列表。
- 可见译文放在源文件旁，隐藏 metadata/cache 放在 `.vscode-doc-translator-cache/`。
- 使用 SHA-256 hash 检测源文档是否过期，以及译文是否被用户编辑。
